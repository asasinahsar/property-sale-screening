# 運用方針定義

対象: 不動産売却スクリーニング（property-sale-screening） / Sprint 3 / Phase1（デモ）

## 技術スタック

| レイヤー | 技術 | 備考 |
|---------|------|------|
| フロントエンド | Next.js | Vercel にデプロイ（フロントのCDN/配信はVercelが担う） |
| バックエンド | FastAPI | AWS ECS（Fargate）。ALB 経由で公開 |
| データベース | PostgreSQL | AWS RDS |
| LLM | Open Responses API + 自律Agent | ReActパターン、マルチステップ推論（スコアリング/定性シグナル抽出/自然言語検索で利用） |
| CI/CD（新規） | GitHub Actions | 自動テスト・自動デプロイ（OIDC + 専用IAMロール） |
| IaC（新規） | Terraform | インフラをコードで管理（state: S3 + DynamoDB） |
| インフラ（新規） | AWS（ECS, RDS, ECR, VPC, S3, ALB） | 本番環境（prod）。ECR にバックエンドイメージを保管 |
| 監視（新規） | CloudWatch, Sentry | ログ・メトリクス・エラー追跡 |

> **AI Agent**: バックエンド（FastAPI/ECS）内で Open Responses API を用いた自律Agent（ReAct）を実行する。一括スクリーニングのスコアリング、出典付き定性シグナル抽出（原文照合）、自然言語検索の条件抽出に用いる。処理要件の詳細は [`../requirements-v1/scoring-and-pipeline.md`](../requirements-v1/scoring-and-pipeline.md) を参照。

|------|--------|------|
| PM | 研修者本人 | 1 |
| フロントエンド | 研修者本人（兼任） | - |

**合計**: 1名（フルスタックで全工程を担当）

> 1名体制のため、レビューは自己レビュー＋CIチェックで品質を担保する。役割は便宜上の区分で、すべて研修者本人が兼任する。

## GitHub ブランチ戦略

**採用戦略**: Trunk-based Development

### ブランチ構成

| ブランチ | 役割 |
|---------|------|
| `main` | 常にデプロイ可能なトランク。短命ブランチをここへ統合 |
| `feature/*` | 機能・修正用の短命ブランチ（数日以内にmainへマージ） |

### 命名規則

- feature ブランチ: `feature/{issue番号}-{機能名}`（例: `feature/12-company-ranking`）
- コミットメッセージ: Conventional Commits（例: `feat: 企業ランキング表示を追加`）

### PR マージ条件

- レビュー承認: 1名体制のため**自己レビュー必須**（PRで差分を確認してからマージ）
- CI チェック: **通過必須**（lint・テスト・ビルド）
- その他: mainへの直接pushは禁止し、必ずPR経由。短命ブランチで頻繁に統合する

## デプロイ・インフラ方針

**デプロイ**: あり

**デプロイ先**: AWS

<!-- cloud-provider: aws -->

### 環境構成

| 環境 | 用途 | 主要サービス | スペック |
|------|------|-------------|---------|
| dev | ローカル開発・単体確認 | ローカル（Docker Compose 等） | 開発者ローカル環境 |
| prod | デモ・本番相当 | フロント=Vercel(Next.js) / バック=ECS Fargate(FastAPI) / RDS(PostgreSQL) / ECR / S3 / ALB | シングルAZ・最小構成（コスト優先） |

> 1名・Phase1デモのため stg（ステージング）は省略し、dev（ローカル）→ prod の2環境とする。需要が出た段階で stg を追加する。
> **フロントエンド**は Vercel にデプロイ（Next.js）。**バックエンド**(FastAPI)は ECS Fargate で稼働し ALB 経由で公開。Vercel から ALB のAPIエンドポイントを呼び出す。
> バックエンドのコンテナイメージは **ECR** に保管。外部開示文書・出力レポート（PDF/Excel/CSV）の保管に **S3** を使用。スコアリングのバッチ処理（自律Agent）は ECS タスク（オンデマンド/スケジュール実行）で実施する想定。

### SLA と可用性

| 項目 | 設定値 |
|------|--------|
| 目標稼働率（SLA） | 99.9%（年間8.7時間ダウン許容） |
| フェイルオーバー | 手動（Phase1はシングルAZ。Phase2でマルチAZ化を検討） |
| Auto Scaling | なし（社内少人数・低トラフィックのため固定キャパシティ） |

### トラフィック予測

| 指標 | 見込み値 |
|------|---------|
| 同時利用ユーザー | 約4人（事業開発企画部の担当者・責任者） |
| DAU / MAU | 約2〜5人 / 約10人 |
| ピーク時同時アクセス | 数RPS未満（年2回の調査期に小さなピーク） |
| データ転送（ファイル等） | あり（EDINET開示文書の取込、PDF/Excel/CSVの出力）。サイズは小〜中。S3で保管・配信 |

> **想定4ユーザーでも余裕。** ECS Fargate 最小構成（1タスク）＋ RDS シングルインスタンスで、同時数人・数RPS未満は十分処理できる。負荷の主因はユーザー数ではなく一括スクリーニング（対象〜100社のLLM処理＝バッチ）であり、同時ユーザー数には依存しない。
> 利用は年2回の調査期に集中する季節性がある。一括スクリーニング（自律Agentバッチ）はオンデマンド実行で、常時高負荷ではない。
> ユーザー増・可用性向上が必要になれば、ECSタスク数の追加（水平スケール）と RDS マルチAZ化で対応可能（Phase2検討）。

### ネットワーク分離（AWS: VPC）

**分離方針**: 同一VPC内でサブネット分離（少人数・低コスト優先）

| サブネット | 配置コンポーネント |
|-----------|-----------------|
| パブリック | ALB（FastAPI/ECS の公開エンドポイント。Vercel のフロントからアクセス） |
| プライベート | ECS Fargate（FastAPI・スコアリングバッチ）/ RDS(PostgreSQL) |

> フロントエンドは Vercel（AWS外）にホストされるため、ALB は Vercel からのアクセスを受ける公開APIとなる。S3・ECR は VPC エンドポイント／マネージドサービスとして接続。RDS はプライベートサブネットに隔離し、直接公開しない。

**環境間通信**: 不要（dev はローカル、prod のみクラウド）

### アクセス権限（IAM）分離戦略

| 対象 | 権限方針 |
|------|----------------------|
| アプリ実行（ECS タスク） | 最小権限を個別付与する IAM ロール（S3読書き・RDS接続・必要なシークレット取得のみ） |
| CI/CD（GitHub Actions） | OIDC + 専用 IAM ロール（アクセスキーは発行しない） |
| 開発者（dev 環境） | 書き込み可（ローカル中心） |
| 開発者（prod 環境） | 原則読み取り専用。デプロイは CI/CD 経由のロールで実施 |

> 外部API資格情報（EDINET等）やDB接続情報は Secrets Manager / SSM Parameter Store に保管し、ハードコードしない。

### IaC

**ツール**: Terraform
**state 管理**: S3 + DynamoDB（ロック用）
**管理対象**: VPC/サブネット、ECS(Fargate)/ALB、RDS、ECR、S3、IAM、CloudWatch 等の AWS リソース（Vercel はプロジェクト側で管理）

### 監視・ロギング

| 項目 | 採用技術 | 用途 |
|------|---------|------|
| ログ・メトリクス | CloudWatch | ECS/ALB/RDS のログ・メトリクス収集、アラーム（CPU/メモリ/エラー率） |
| エラー追跡 | Sentry | フロント(Next.js)・バック(FastAPI)の例外・エラーの捕捉と通知 |
| 主要監視対象 | スクリーニング実行（バッチ）の成否・所要時間、API エラー率、RDS 接続/容量、LLM(Open Responses API) 呼び出しの失敗 | 異常系の早期検知（異常系の共通方針＝エラー表示＋再試行と連動） |

> Phase1 は最小構成（基本メトリクス＋エラー通知）。Phase2 でダッシュボード・SLOアラート・通知連携を拡充する。

## リリース頻度と承認フロー

**リリース頻度**: 隔週（スプリント単位）

### 通常リリースフロー

1. feature ブランチで実装 → PR作成、GitHub Actions の CI（lint/テスト/ビルド）通過
2. 自己レビューで差分確認のうえ main へマージ
3. main へのマージで CI/CD が自動デプロイ：
   - フロント（Next.js）→ **Vercel** へデプロイ
   - バック（FastAPI）→ イメージを **ECR** にpush → **ECS Fargate** をローリング更新（必要に応じ RDS マイグレーション）
4. デプロイ後にデモ画面で動作確認（スモークテスト）。CloudWatch/Sentry で異常がないか確認

**最終承認者**: 研修者本人（PM兼任）
**承認タイミング**: スプリント末。CI通過＋動作確認をもってリリース可

### 緊急リリース（ホットフィックス）フロー

1. `feature/hotfix-*` ブランチで最小修正 → CI通過後 main へマージ
2. prod へ即時デプロイ → 動作確認

**通知先**: 研修内の報告チャネル（Slack 等。1名体制のため記録目的で残す）
