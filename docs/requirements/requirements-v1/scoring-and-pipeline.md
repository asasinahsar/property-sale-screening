# 処理要件: スコアリング & データパイプライン（非画面・サポート用）

本ドキュメントは画面ではなく、各画面（P001〜P004）の Input となる「指標・スコア・定性シグナル・確信度」の**生成ロジックを概念レベルで定義**する。各画面はここで生成された結果を受け取る前提とする。

> **スコア呼称**: 構造スコア（=定量）／イベントスコア（=定性）／総合スコア（両者を統合した派生値・ランキングの並び順キー）。
> **粒度の注記**: 本フェーズは概念レベル。計算式の係数・重み・テーブル・API仕様は次のDB/API設計フェーズで詳細化する。

## データソースと母集団（Phase1）

| 項目 | 内容 |
|------|------|
| EDINET API | 有価証券報告書、適時開示、大量保有（変更）報告書を取得 |
| YFinance | 株価・時価総額等の市場データ |
| 母集団 | 賃貸等不動産の時価開示があり、一定の財務条件を満たす上場企業。デモは約100社をシードとし、事前バッチで投入（取込専用画面は持たない） |

## 構造スコア（定量）の算出要素（Phase1）

- 含み益／時価総額（含み益倍率）
- 修正PBR（純資産に含み益を加味して再計算）
- 自己資本比率
- 対象資産のROIC 対 WACC
- **正規化**: 業種内z-score（業種ごとの水準差を正規化）で各指標を相対化してから合成（0–100）

## イベントスコア（定性）の算出要素（Phase1）

- 定性シグナルの種別×強度の重み付け（例：アクティビストの株主提案 ＞ 中計の資本効率目標明記 ＞ 一般的な見直し）
- 大量保有報告書からの既知アクティビスト保有／買い増しの検知を**先行シグナル**として加点
- 反証シグナル（売却否定）の減点
- シグナルの新しさ（recency）を考慮

## 定性シグナル抽出（LLM）の要件（Phase1）

- **出力スキーマを固定**：シグナル種別・強度・document_type・source_page・引用文を必須項目とする
- **原文照合**：抽出した引用文が出典原文に実在するか検証し、出典が取れないシグナルは不採用（ハルシネーション対策＝トレーサビリティ要件の担保）

## 統合スコアリングと確信度（Phase1）

- 構造スコア × イベントスコア から総合スコアを算出（並び順キー）
- AI総合判定コメントを生成し、参照した根拠に紐づける
- **確信度／データ充足度**を算出（開示の量・新しさに基づく）し、各画面へ提供（P001の低確信度マーク、P002の確信度表示の入力）

## 精度検証・改善

| 項目 | 内容 | Phase |
|------|------|-------|
| バックテスト | 過去に不動産売却を実施した企業でスコアの事前妥当性を検証し、しきい値・重みをキャリブレーション | Phase1（検証活動） |
| ヒューマン・イン・ザ・ループ | 担当者の当たり/外れラベルを蓄積し重みを再調整する学習ループ | Phase2（定常運用） |

## 各画面への供給データ（参照早見）

| 供給先 | 主な供給データ |
|--------|--------------|
| P001 ダッシュボード | 構造/イベント/総合スコア、KPI集計、直近イベント、イベントブースト倍率、確信度 |
| P002 企業詳細 | スコア内訳、定量指標標準セット、ROIC/WACC、定性シグナル（支持/反証）、出典、確信度、AI総合判定 |
| P003 ロングリスト | 各スコア、含み益・倍率、出典サマリ |
| P004 通知・ウォッチリスト（Phase2） | スコア更新、新規開示・大量保有報告書の検知イベント |
