## 0. プロジェクト概要（30秒サマリー）

三菱地所「事業開発企画部」向けのデモアプリ。企業の**有価証券報告書（財務データ）**と**中期経営計画・IRリリース（テキスト情報）**をLLMで分析し、**不動産売却可能性の高い企業を自動スクリーニング**する。定量・定性の二系統で処理した結果を統合スコアリングし、**根拠サマリー（出典付き）付きのランキング**として出力する。

- **発注/活動文脈**: Boost Consulting による「BxS活動支援」の一環。三菱地所のAI活用を「理解浸透フェーズ」から「事業インパクト創出フェーズ」へ転換する目玉ユースケースの1つ。
- **Sprint3 の評価主軸**: 「オンラインで動き続けるサービスとして、いかに頑健に作られているか（＝非機能要件）」。Sprint3 発表は **6月30日(火)**。
- **データソース**: **EDINET API** と **YFinance API** を使えば企業のIR情報を無料取得できる（これを活用する）。
- **デモURL（参考実装）**: https://mitubishidemo-production.up.railway.app/

## 3. As-Is と To-Be

（出典: As-Is/To-Be 図 = MITSUBISHI ESTATE 作成スライド）

- **対象業務**: 売却可能性のある不動産を洗い出しアプローチするため、企業の決算情報や不動産情報をリサーチする。
- **ユースケース概要**: AI活用による、不動産売却可能性のある企業のスクリーニング、及びスクリーニングした企業の決算資料の読込を効率化。

### As-Is（現状）

不動産売却可能性のある企業を探るため、部内のリサーチ負荷が高い（**年間500時間**）。

- 候補企業の抽出・スクリーニング: 事開担当が**年2回**、財務数値を見ながら**SPEEDA等**で有力企業を調査 → 調査対象企業のスクリーニング → 深掘り対象ロングリスト
- 決算・不動産情報の深掘り: 事開担当が**年2回・約50社程度**のIR資料等を読み込み（上記スクリーニング業務を含め**年500時間**）→ 決算・不動産情報調査 → 売却可能性のある不動産

要件ブリーフ補足:
- 調査対象企業の選定基準が**属人的**で、担当者によって着眼点にばらつきがある。
- 財務数値の確認とテキスト情報のリサーチが**別々の作業**となっており、統合的な判断に時間がかかる。
- **→「人手によるリサーチ」と「属人的な判断」が非効率と品質ばらつきを生んでいる。**

### To-Be（目指す姿）

**年間約499時間の業務時間削減を目標**（※AIが数値を扱うのが苦手なため、人での確認は一定残る想定。やや挑戦的なためPoCを通して精緻化）。

- 候補企業の抽出・スクリーニング: **AIが**調査対象企業のスクリーニングを自動化 → 深掘り対象ロングリスト → 事開担当が**確認/修正**（財務数値を加味した調査のため、人目の確認が残る可能性有）
- 決算・不動産情報の深掘り: **AIが**決算・不動産情報調査を自動化 → 売却可能性のある不動産 → **確認/修正**

要件ブリーフ補足:
- EDINET/XBRLの財務データと、中計・IRリリース等のテキスト情報を自動で収集・分析できる。
- 定量・定性の両面から統合スコアリングされた企業ランキングが自動生成される。
- AI判定の全根拠（出典ページ・引用文）がトレース可能な状態で提示され、担当者は「根拠確認」ではなく「追加観点の検討」に集中できる。
- 継続モニタリングにより、新規開示情報があれば随時スコアが更新される。
- **→ AIが一次スクリーニングを担い、担当者は深掘り調査と最終判断に専念できる。**

## 5. ソリューション全体像

- **Phase 1** → 定量・定性パイプラインを構築し、統合スコアリングの精度を技術検証
- **Phase 2** → 継続モニタリング機能を追加し、本番運用へ
- **EDINET API** と **YFinance API** を使えば、企業のIR情報を無料で取得できる。このデータを活用する。

```
入力データ
  ├── 定量パイプライン（財務データ）
  │     EDINET/XBRL 有価証券報告書
  │       │
  │       ▼
  │     専用パーサーで構造化抽出
  │       │
  │       ▼
  │     定量スクリーニング（PBR、自己資本比率、賃貸等不動産時価 等）
  │
  └── 定性パイプライン（テキスト情報）
        中計・適時開示・IRリリース
          │
          ▼
        LLMでシグナル抽出・分類
          │
          ▼
        定性シグナル検出（アクティビスト動向、不動産売却示唆 等）
          ↓ 両パイプラインを統合
┌─────────────────────────────────────┐
│ 統合スコアリング                      │
│  ・定量スコア × 定性シグナル数で総合評価  │
│  ・根拠サマリーを自動生成             │
│  ・全根拠に出典（ページ・引用文）を付与   │
└──────────────┬──────────────────────┘
               ▼
┌─────────────────────────────────────┐
│ アウトプット                          │
│  ・企業ランキング（スコア順）           │
│  ・根拠レポート（出典付き）            │
│  ・継続モニタリング（新規開示の検知）    │
│        ↓                            │
│  担当者確認 → 深掘り調査へ            │
└─────────────────────────────────────┘
```

二系統パイプライン図（提案資料 p.9）の要旨:
- 財務データは専用パーサーで正確に抽出、テキスト情報（中計・IR）はLLMでシグナル検出。二系統を統合スコアリングし、根拠サマリー付きでランキングを自動生成。
- 定量パイプライン: EDINET/XBRL有価証券報告書 → 専用パーサーで構造化抽出 → 定量スクリーニング → 企業ランキング
- 定性パイプライン: 中計・適時開示IRリリース → LLMでシグナル抽出・分類 → 定性シグナル検出
- 統合スコアリング（根拠サマリー自動生成） → 根拠レポート／継続モニタリング → 担当者確認 → 深掘り調査へ
- **※ 全ての監査観点に出典を自動付与するため、担当者は根拠確認ではなく追加観点の検討に集中可能。**

## 6. デモアプリ：MVP機能と画面イメージ

### MVP機能

- **定量スクリーニング機能（バックエンド）**
  - 財務CSVを読み込み、PBR・自己資本比率・賃貸等不動産時価などの指標で定量スコアを算出
- **定性シグナル検出機能（バックエンド）**
  - IRテキストをLLM（Claude/OpenAI API）に渡し、売却シグナルの有無・種別・強度を抽出
  - 出典（document_type・source_page・引用文）を自動付与
- **統合スコアリング（バックエンド）**
  - 定量スコア × 定性シグナル数で総合スコアを算出
  - AI総合判定コメントを自動生成
- **ダッシュボード（フロントエンド）**
  - 企業ランキング一覧（総合スコア順）
  - 業種・PBR・シグナル数でのフィルタリング
- **企業詳細画面（フロントエンド）**
  - 定量分析パネル（財務指標一覧）
  - 定性シグナルパネル（シグナル種別・出典・引用文）
  - AI総合判定テキスト

### 1. 背景／本プロジェクトの目的

昨年度BxSの取り組みおよび次年度の活動目標:

## 付録A. 非機能要件PDF 全文（6軸の詳細）

（出典: `非機能要件について.pdf`。R2B研修 (Sprint3)、2026年3月16日、boostconsulting）

**1. 可用性・信頼性 (Availability & Reliability)** — サービスを停止させず、ユーザーがいつでも使える状態を維持できているか？
- 稼働率: 求められる停止許容時間の定義と実現
- 障害耐性: 一部のサーバーやDBがダウンしても、自動で切り替えてサービスを継続できる冗長化の仕組み
- バックアップ・復旧: データ消失を防ぎ、障害時に迅速に復旧する手順

**2. 性能・拡張性 (Performance & Scalability)** — 大量アクセスやデータ増加時でも、快適な動作速度を維持できているか？
- レスポンス速度: 画面遷移や処理結果表示までの時間（レイテンシ）の最小化
- スケーラビリティ: ユーザー増加に合わせて柔軟に拡張（スケールアップ/アウト）できる設計
- 負荷テスト: 想定最大アクセス数をクリアできるかの事前検証

**3. セキュリティ (Security)** — ユーザーデータや企業の機密情報を守り、信頼を担保できているか？
- 認証・認可: 安全なログインと、適切なアクセス権限管理
- 通信・データの暗号化: 通信の暗号化（SSL/TLS）や、DB上の個人情報の暗号化
- 脆弱性対策: SQLインジェクション、XSS、環境変数のハードコード等の排除

**4. 運用・保守性 (Maintainability & Supportability)** — リリース後、効率的にメンテナンスを行い、長期的にサービスを存続させることができているか？
- 監視耐性: サーバー負荷、エラーログ、異常なアクセスをリアルタイムに監視する仕組み
- リリース管理: 新機能追加やバグ修正の際、サービスを止めずに更新できる仕組み（デプロイフロー）
- ログ記録: トラブル時に原因特定が容易になるよう、操作やエラーの履歴（ログ）を適切に保存・管理する

**5. ユーザビリティ・アクセシビリティ (Usability & Accessibility)** — 誰でもストレスなく利用できるシステムになっているか？
- ユーザー体験(UX): 直感的なUIや操作のしやすさ
- レスポンシブ対応: PC、スマホ、タブレットなど、様々なデバイスで適切に表示されること

**6. コンプライアンス・法規制対応 (Compliance)** — 法令や業界の基準に適合しているか？
- 個人情報保護（GDPR, CCPA等）: ユーザーデータの取り扱いや削除要請への対応
- 特定商取引法・資金決済法: オンライン決済サービスにおける法律遵守

## 付録C. 参考リサーチ（市場背景・指標・データソース）

> ※ この付録は元資料には含まれない**補足リサーチ**です（要件定義の背景補強用）。

### C-1. 市場・ドメイン背景（なぜ今CRE売却が増えているか）

- **東証のPBR改善要請（2023年3月）**: 「資本コストや株価を意識した経営の実現に向けた対応」。プライム市場の約半数がPBR1倍割れ・ROE8%未満が出発点。2024年末時点でプライム企業の約9割が対応開示済み。資本効率改善圧力が、低収益・遊休不動産の売却と含み益顕在化を後押し。
- **アクティビストの急増**: 日本参入は2024年10月時点で73者（5年で倍増）。2025年の株主提案は137議案で過去最多。テーマは政策保有株の縮減・株主還元・不動産売却に集中（例: エリオットが東京ガスに都内一等地の売却を要求）。
- **CRE売却の加速**: 2024年度に国内不動産売却を開示した企業は85社、面積は前年比約1.6倍、9割が譲渡益計上。本社ビルのセール&リースバックも増加。

### C-2. スクリーニング指標の勘どころ

- **PBR = 株価 ÷ BPS（1株純資産）**。1倍割れ＝解散価値割れ＝割安サイン。ただし**簿価ベースのため不動産の含み益は反映されにくい**。「PBRが低い × 賃貸等不動産の含み益が大きい」企業ほど売却圧力がかかりやすい。
- **賃貸等不動産の時価注記**（2010年3月期〜の会計基準, ASBJ）: 有報の注記で簿価と時価が開示され、`時価 − 簿価 = 含み益` が機械的に取得可能。定量パイプラインの中心指標。
- **自己資本比率**が高すぎる＝ROEの重し＝株主還元・資本圧縮の余地。
- **売却シグナルの型**: ROIC < WACC、アクティビストの株主提案、中計の資産効率への言及、政策保有株・遊休地の保有。
- **教科書ケース＝ワキタ（ストラテジックキャピタル）**: 「不動産事業のセグメントROIC平均2.7%＝WACC未満」「高すぎる自己資本比率」「PBR1倍割れ」を指摘し、DOE6%配当・非公開化検討委員会の設置を提案。デモの題材と一致。

### C-3. データソース実現性

- **EDINET API（v2・無料、要APIキー登録）**: 有報・四半期・適時開示のXBRL書類を取得。賃貸等不動産の時価注記から簿価・含み益を抽出。
- **yfinance（無料）**: 株価・時価総額・PBRなど市場系を補完。日本固有の注記は弱い。
- **分担の目安**: 定量＝EDINET（注記含む簿価・時価）／市場系＝yfinance／定性＝LLMでIR・中計のシグナル抽出。
